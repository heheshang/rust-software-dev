# T0 - 需求门禁

## 📋 节点目标
在进入正式开发前，过滤掉「说不清、道不明、做不了」的需求。

## ✅ 完成标准（DoD）
- 得分 ≥ 70 分
- 2 个 P0 项全部通过
- 需求澄清记录完整归档

## 🔍 检查项明细

| 检查项 | 分值 | 级别 | 通过标准 |
|--------|------|------|---------|
| 功能范围明确 | 25 | P0 | 一句话能说清做什么 |
| 用户角色明确 | 20 | P0 | 明确谁在用这个功能 |
| 边界清晰 | 15 | P1 | 明确不做什么 |
| 数据来源明确 | 15 | P1 | 数据从哪来、存哪、怎么用 |
| 非功能需求 | 10 | P2 | 性能、安全、可用性期望 |
| 依赖关系 | 10 | P2 | 是否依赖其他功能/系统 |
| 验收标准 | 5 | P2 | 可量化的完成标准 |

## 🤖 自动执行逻辑

```bash
# 步骤 1：运行评分脚本
./scripts/t0-check.sh <功能名称>

# 步骤 2：根据返回码判断
#   0 = 通过 (≥70分)
#   1 = 不通过 (<70分 或 P0 失败)
#   2 = 有风险 (50-69分，需要用户确认)

# 步骤 3：通过后自动归档
./scripts/archive-node-docs.sh T0 <功能名称>
```

## 📦 归档产出物
- `docs/prd/T0-Checklist-YYYYMMDD.md` - 检查评分结果
- `docs/prd/Requirements-Clarification-YYYYMMDD.md` - 需求澄清记录

## ⚠️ 常见失败场景
1. **「做一个用户中心」** → 范围太大，P0 不通过
2. **「管理员用的」** → 角色太模糊，P0 不通过
3. **「没有不做的，全做」** → 边界不清晰，P1 扣分
