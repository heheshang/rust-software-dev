# T1 - PRD 输出

## 📋 节点目标
把模糊的需求变成清晰、可执行、可测试的产品需求文档。

## ✅ 完成标准（DoD）
- PRD 所有字段填写完整
- Gherkin 场景 ≥ 3 个
- 验收标准清晰、可量化
- 边界和异常场景有定义

## 🔍 PRD 检查项

| 检查项 | 级别 | 通过标准 |
|--------|------|---------|
| 功能背景 | P0 | 为什么做这个功能，解决什么问题 |
| 用户角色 | P0 | 明确谁在用，每个角色的权限 |
| 功能说明 | P0 | 清晰的功能流程图 |
| 数据来源 | P1 | 数据从哪来、存哪、怎么用 |
| 边界定义 | P1 | 明确不做什么 |
| 异常场景 | P1 | 错误处理说明 |
| 验收标准 | P2 | 可量化的完成定义 |

## 🤖 自动执行逻辑

```bash
# 步骤 1：运行 PRD 完整性检查
./scripts/t1-prd-check.sh

# 步骤 2：检查通过后，等待 PM 确认
# 人工评审 → 签署确认
```

## 🔙 回滚触发条件
- 核心功能描述缺失
- 边界完全没定义
- 技术可行性有本质疑问

回退到 **T0**，重新做需求澄清。

## 📦 归档产出物
- `docs/prd/PRD-{Feature}-YYYYMMDD.md`
- `docs/prd/Gherkin-Scenarios-YYYYMMDD.md`
- `docs/prd/PRD-Review-Notes-YYYYMMDD.md`

## ⚠️ 常见陷阱
1. **写解决方案而不是需求** - 「用 Redis 缓存」是解决方案，不是需求
2. **太笼统** - 「用户体验好」不是验收标准，「页面加载 < 200ms」才是
3. **没有边界** - 什么都想做，结果什么都做不好
