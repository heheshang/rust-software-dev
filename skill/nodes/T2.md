# T2 - 架构设计

## 📋 节点目标
确定技术方案、API 契约、数据模型，让前后端可以并行开发。

## ✅ 完成标准（DoD）
- ADR 架构决策记录已签署
- OpenAPI 3.0 Spec 已完成并评审
- 数据库 ER 图已确定
- RFC（如有重大变更）已通过 Accept 状态
- 架构健康评分 ≥ 80 分

## 🔍 架构检查 5 维度

| 维度 | 权重 | 检查脚本 |
|------|------|---------|
| 依赖方向检查 | 30% | `check_dependency_direction()` |
| 循环依赖检测 | 25% | `check_cyclic_dependency()` |
| 模块边界完整性 | 20% | `check_module_boundary()` |
| 代码复杂度分析 | 15% | `check_complexity_and_debt()` |
| 技术债务统计 | 10% | 统计 FIXME/TODO/HACK 数量 |

## 🤖 自动执行逻辑

```bash
# 步骤 1：运行架构健康检查
./scripts/architecture-health-check.sh

# 步骤 2：读取评分
# < 60 分 → 🔴 不通过，回滚 T1
# 60-79 分 → 🟡 有风险，需要 TechLead 确认
# ≥80 分 → ✅ 通过

# 步骤 3：自动归档架构快照
./scripts/archive-node-docs.sh T2 <功能名称>
```

## 🔙 回滚触发条件
- 架构健康评分 < 60 分
- 发现无法解决的技术难点
- 需求理解有本质偏差

回退到 **T1**，重新澄清需求后再做设计。

## 📦 归档产出物
- `docs/architecture/ADR-XXX.md` - 架构决策记录
- `docs/api/OpenAPI-3.0-spec.yaml` - API 契约
- `docs/architecture/ER-Diagram.png` - ER 图
- `docs/architecture/Architecture-Report-YYYYMMDD.md` - 架构健康报告

## ⚠️ 常见陷阱
1. **过早优化** - 还没理解需求就开始想性能
2. **过度设计** - 为 1% 的场景做 99% 的架构
3. **不写 ADR** - 决策只在脑子里，后人无法理解
