## 🎯 设计原则

### 1. **单一职责**
| 目录 | 职责 | 修改频率 |
|------|------|---------|
| `meta/` | 全局常量配置 | 极低（项目初始定了就不改） |
| `nodes/` | 每个节点的定义 | 低（节点标准稳定） |
| `checks/` | 质量检查的阈值和权重 | 中（根据实际情况调优） |
| `rollback/` | 回滚规则 | 低 |
| `archive/` | 归档标准 | 低 |

### 2. **正交设计**
- ✅ 改检查阈值不会影响节点定义
- ✅ 改回滚规则不会影响归档标准
- ✅ 每个模块可以独立测试、独立迭代

### 3. **可插拔架构**
想换掉架构检查脚本？
```
只需要改：  checks/architecture.yaml
不需要改： nodes/T2.md
不需要改： skill/SKILL.md
不需要改： rollback/matrix.yaml
```

## 🚀 如何使用

### Hermes Agent 加载
```bash
# 自动加载整个 skill 目录
hermes load skill ./skill

# 启动流水线
hermes start pipeline user-auth
```

### 单独测试某个节点
```bash
# 只测试 T0 节点逻辑
hermes test node T0

# 只运行架构健康检查
hermes test check architecture
```

### 修改某个配置
```bash
# 调整架构健康的权重（不需要改代码）
vim skill/checks/architecture.yaml
```

## 📊 模块化 vs 单体文件 对比

| 维度 | 大单体文件（之前） | 模块化架构（现在） |
|------|-------------------|------------------|
| **可读性** | ❌ 5000行翻死人 | ✅ 每个文件 < 100 行 |
| **可维护性** | ❌ 改一处怕影响全局 | ✅ 正交设计，影响范围可控 |
| **可测试性** | ❌ 无法单独测试 | ✅ 每个节点可以单独测试 |
| **多人协作** | ❌ 容易冲突 | ✅ 每人改自己的模块，无冲突 |
| **版本管理** | ❌ git blame 一团糟 | ✅ 每个模块变更历史清晰 |
| **新人上手** | ❌ 看一周才能懂 | ✅ 从 SKILL.md 入手，逐层深入 |
| **调试排错** | ❌ 大海捞针 | ✅ 出问题只看对应模块 |

## 💡 最佳实践

### 1. **修改前先想清楚改哪个模块**
- 想改「回滚规则」→ 改 `rollback/matrix.yaml`
- 想改「架构检查阈值」→ 改 `checks/architecture.yaml`
- 想改「T2 完成标准」→ 改 `nodes/T2.md`
- 想改「超时时间」→ 改 `meta/definitions.yaml`

### 2. **不要跨模块写逻辑**
❌ 错误：在 `nodes/T2.md` 里写回滚规则
✅ 正确：回滚规则统一写在 `rollback/matrix.yaml`

### 3. **配置驱动，不要硬编码**
所有阈值、权重、超时时间都放在配置文件，绝对不要在代码里写死！
