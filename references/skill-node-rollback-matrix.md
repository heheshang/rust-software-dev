# 🎯 SKILL 节点级回滚矩阵 v1.0

> **核心设计原则**：每个阶段节点都必须定义：
> 1. ✅ 怎么判断「这个节点完成了」（DoD - Definition of Done）
> 2. ❌ 怎么发现「这个节点出错了」（检测方式）
> 3. ↩️ 发现出错后「回退到哪个节点」（精确回滚路径）
> 4. 🔄 回滚后「怎么重新进入流程」（重入机制）

---

## 📋 T0-T9 节点回滚完整矩阵

| 阶段节点 | ✅ 完成标准 (DoD) | ❌ 发现错误的方式 | ↩️ 出错时回退到 | 🔄 重入机制 | ⏱️ 预计返工耗时 |
|---------|------------------|------------------|----------------|-----------|---------------|

| **T9 最终评审** | 架构评分 ≥ 80，无 P0 问题，签字确认 | 最终评审发现架构腐化 / 安全漏洞 | **回退到对应问题源节点** | 定位问题根源，修复后重新走完整流程 | 1 天 - 1 周 |

## 🔍 每个节点的详细回滚规则

### 🔴 T0 需求门禁节点

**✅ 完成标准**：
- T0 检查脚本输出 `🟢 通过`
- 得分 ≥ 70 分
- P0 项（功能范围、用户角色）全部通过

**❌ 发现错误的方式**：
1. T0 脚本直接输出 🔴 不通过
2. 用户在 T1 阶段补充了重要需求信息
3. TechLead 在 T2 发现需求理解有本质偏差

**↩️ 回滚路径**：
- **不跨阶段回退**，就停在 T0
- 不需要撤销任何已完成工作（因为还没开始）

**🔄 重入机制**：
```bash
# 重新运行 T0 检查脚本
./scripts/t0-check.sh  # （需要把 SKILL 中的脚本提取出来）

# 补充需求信息后，再次运行直到通过
```

**⚠️ 常见出错场景**：
- 「我以为这个功能包含 XX，结果没做」→ 回 T0 重新明确范围
- 「其实还有另一种用户角色」→ 回 T0 补充角色说明

---

### 🟡 T1 PRD 输出节点

**✅ 完成标准**：
- PRD.md 所有字段已填写
- Gherkin 场景 ≥ 3 个
- 验收标准清晰可量化

**❌ 发现错误的方式**：
1. TechLead 在 T2 架构设计时，发现 PRD 有逻辑矛盾
2. 开发在 T3 设计时，发现 PRD 缺少关键业务规则
3. 产品在 T5.5 走查时，发现理解和 PRD 不一致

**↩️ 回滚路径**：
- **回退到 T0**，重新做需求澄清
- T2/T3 的工作暂停，直到 PRD 修正完成

**🔄 重入机制**：
```bash
# Step 1：更新 PRD 文档
vim docs/prd/PRD-XXX.md

# Step 2：重新走 T0 评分
./scripts/t0-check.sh

# Step 3：通知所有相关人员需求变更
echo "PRD 已更新，请重新确认理解"
```

**⚠️ 常见出错场景**：
- 「这个业务规则其实是这样的...」→ PRD 没写清楚
- 「还有这个异常情况也要处理」→ 边界场景遗漏

---

### 🔴 T2 架构设计节点（最关键的回滚点）

**✅ 完成标准**：
- ADR.md 架构决策记录已签署
- OpenAPI Spec 3.0 已完成
- 数据库模型 ER 图已评审
- RFC（如果需要）已通过 Accept 状态

**❌ 发现错误的方式**：
1. 开发在 T3 发现技术方案不可行
2. T4.5 联调发现 API 设计有本质缺陷
3. T6 性能测试发现架构选型错误
4. T9 最终评审发现架构腐化根源在 T2

**↩️ 回滚路径**：
- **回退到 T1**，重新做方案选型
- **所有在 T2 之后的工作全部暂停**
- 如果已经进入 T4 开发，损失可能很大

**🔄 重入机制**：
```bash
# Step 1：创建新的 ADR 记录
cp docs/architecture/ADR-001.md docs/architecture/ADR-001-REVISED.md

# Step 2：更新 RFC 状态为 Discuss（重新讨论）
./scripts/rfc-status.sh --update RFC-XXX status=Discuss

# Step 3：重新走架构评审流程
./scripts/architecture-health-check.sh
```

**⚠️ 最容易造成重大返工的节点**：
- T2 设计的数据库表结构有问题 → T4 写了一堆代码全错
- T2 设计的 API 不符合前端实际使用 → T4.5 联调全部翻车
- T2 选错了技术方案 → T6 性能完全不达标

**💡 预防措施**：T2 完成后强制做一次「方案可行性预演」，不要等到写了很多代码才发现问题。

---

### 🟡 T3 技术设计节点

**✅ 完成标准**：
- 后端 TechDesign.md 完成
- 前端 TechDesign.md 完成
- 前后端都评审通过

**❌ 发现错误的方式**：
1. 写代码时发现设计的方案在技术上不可行
2. Code Review 发现实现和设计不一致
3. 测试时发现设计没有考虑边界情况

**↩️ 回滚路径**：
- **回退到 T2**（如果是架构级问题）
- **停在 T3 修正**（如果是局部设计问题）

**🔄 重入机制**：
```bash
# 如果是局部问题：
# 1. 更新 TechDesign.md
# 2. 通知相关人员设计变更

# 如果是架构级问题：
# 1. 回 T2 重新做架构设计
# 2. 重新评审后再进入 T3
```

---

### 🟠 T4 并行开发节点

**✅ 完成标准**：
- 前端：自测通过，ESLint 无 error，TypeScript 无 error
- 后端：cargo test 全部通过，clippy 无 warning
- 前后端各自的 Feature 分支 CI 绿灯

**❌ 发现错误的方式**：
1. CI 测试失败
2. Code Review 发现严重问题
3. T4.5 联调发现大量不匹配

**↩️ 回滚路径**：
- **前端回 T3 前端**，**后端回 T3 后端**
- 不需要跨回滚到 T2（除非是设计问题）

**🔄 重入机制**：
```bash
# 前端修复
cd frontend
npm run type-check  # 确保零错误
npm run test:unit   # 确保测试全过

# 后端修复
cd backend
cargo clippy -- -D warnings
cargo test

# 修复完成后，重新提交 PR
```

**⚠️ 常见场景**：
- 前端写了一堆 any 类型 → 回 T4 修正，不能流入 T4.5
- 后端有未处理的 unwrap() → 回 T4 修正，不能流入 T5

---

### 🔴 T4.5 契约校验节点（第二关键回滚点）

**✅ 完成标准**：
- contract-validation.sh 输出 ✅ 通过
- API 契约一致性 ≥ 90%
- 前端类型从 Spec 重新生成无误

**❌ 发现错误的方式**：
1. 契约检查脚本直接报 🔴
2. 联调时发现实际返回和 Spec 不一致
3. 前端类型生成失败

**↩️ 回滚路径**：
- **前后端各自回 T4 修复**
- 如果是 Spec 本身设计问题 → **回 T2 修正 Spec**

**🔄 重入机制**：
```bash
# Step 1：运行契约检查，定位问题
./scripts/contract-validation.sh

# Step 2：如果是 Spec 问题
vim docs/api/OpenAPI-3.0-spec.yaml

# Step 3：重新生成前端类型
npx openapi-typescript docs/api/OpenAPI-3.0-spec.yaml -o frontend/src/types/api.generated.ts

# Step 4：后端更新路由实现

# Step 5：重新跑契约检查，直到通过
```

**⚠️ 这个节点的重要性**：
T4.5 是前后端第一次「实际握手」，如果这里发现大量不匹配，说明 T2 的 Spec 设计有问题，或者 T4 的实现严重偏离。这个节点不通过，绝对不能进 T5。

### 🟠 T5 后端完成节点

**✅ 完成标准**：
- 所有 API 接口在 Postman 可正常调用
- cargo test --all 全部通过
- migration-safety-check.sh 无 P0 问题

**❌ 发现错误的方式**：
1. 集成测试失败
2. Migration 检查发现危险操作
3. 性能测试发现慢查询

**↩️ 回滚路径**：
- **回 T4 后端开发**，修复问题
- 如果是数据模型设计问题 → **回 T2 重新设计**（最糟糕的情况）

**🔄 重入机制**：
```bash
# Step 1：跑 Migration 安全检查
./scripts/migration-safety-check.sh

# Step 2：如果有 P0 问题，重写 Migration
cd backend/migration
# 修正 SQL

# Step 3：重新跑所有测试
cargo test --all

# Step 4：再次跑 Migration 检查，确认无 P0
```

### 🟡 T5.5 UI 走查节点

**✅ 完成标准**：
- UI 走查清单 P0 项全部通过
- 还原度 ≥ 90%
- 产品 / UI 设计师签字确认

**❌ 发现错误的方式**：
1. 走查时发现和设计稿差异太大
2. 交互体验不符合预期
3. 响应式适配有问题

**↩️ 回滚路径**：
- **回 T4 前端**，修复样式和交互
- 不需要回滚后端（除非是数据问题导致的显示异常）

**🔄 重入机制**：
```bash
# Step 1：前端修复样式/交互

# Step 2：重新部署测试环境

# Step 3：再次走查，只检查上次没通过的项

# Step 4：走查通过，签字确认
```

**⚠️ 注意**：UI 走查不通过，绝对不能进 T6 QA 测试。否则 QA 会报一堆样式 Bug，浪费 QA 时间。

### 🔴 T6 QA 测试节点（第三关键回滚点）

**✅ 完成标准**：
- P0 Bug = 0
- P1 Bug ≤ 3
- 核心流程测试通过率 = 100%
- 安全扫描无 Critical/High 漏洞

**❌ 发现错误的方式**：
1. QA 测试报告有 P0 Bug
2. 安全扫描发现 Critical 漏洞
3. 性能不达标

**↩️ 回滚路径**：
- **Bug 回退到对应负责人**（前端 Bug 回前端，后端 Bug 回后端）
- 如果是架构级性能问题 → **回 T2 重新设计**

**🔄 重入机制**：
```bash
# Step 1：开发修复 Bug，提交代码

# Step 2：回归测试（至少测修复的 Bug + 相关功能）

# Step 3：重新跑安全扫描
./scripts/security-scan.sh

# Step 4：QA 复测通过后，才能算 T6 完成
```

**⚠️ 这个节点的铁律**：
只要有 1 个 P0 Bug，T6 就不算完成。绝对不能带 P0 Bug 进 T7。

### 🟢 T7 文档节点

**✅ 完成标准**：
- 所有必选文档齐全
- document-quality-check.sh 评分 ≥ 80
- CHANGELOG.md 已更新

**❌ 发现错误的方式**：
1. 文档质量检查脚本报警
2. T9 评审发现文档过时/缺失

**↩️ 回滚路径**：
- **不回滚代码，只补文档**
- 可以和 T6 并行进行

**🔄 重入机制**：

```bash
# Step 1：跑文档质量检查
./scripts/document-quality-check.sh

# Step 2：补充缺失的文档

# Step 3：重新生成文档索引
./scripts/generate-doc-index.sh
```

### 🟠 T8 部署 & 运维节点

**✅ 完成标准**：
- staging 环境部署成功
- healthcheck 全部通过
- 性能压测达标

**❌ 发现错误的方式**：
1. 部署脚本执行失败
2. staging 环境服务启动失败
3. 压测 QPS 不达标

**↩️ 回滚路径**：
- **回 T6 测试环境**，排查问题
- 如果是配置问题 → 修配置重部署
- 如果是代码问题 → 回 T4 修复代码

**🔄 重入机制**：
```bash
# Step 1：检查部署日志
kubectl logs ... / docker logs ...

# Step 2：定位问题（配置？代码？依赖？）

# Step 3：修复后重新部署 staging

# Step 4：再次跑 healthcheck 和压测
```

### 🔴 T9 最终评审节点（最后防线）

**✅ 完成标准**：
- 架构健康评分 ≥ 80
- 安全扫描无 P0 问题
- 所有 RFC 已归档或关闭
- 所有文档齐全
- TechLead + 产品 + QA 三方签字

**❌ 发现错误的方式**：
1. 架构健康评分 < 70
2. 发现 P0 安全漏洞
3. 发现核心功能没做 / 不符合需求

**↩️ 回滚路径**：
- **回退到问题对应的源头节点**
- 架构问题 → 回 T2
- 代码质量问题 → 回 T4
- 安全问题 → 回 T6
- 文档问题 → 回 T7

**🔄 重入机制**：
```bash
# Step 1：定位问题根源在哪个阶段
# 是 T2 设计就错了？还是 T4 实现错了？

# Step 2：回退到对应节点修复

# Step 3：从那个节点开始，重新走完整的后续流程
# （注意：不能跳过任何节点！）

# Step 4：全部走完后，再次做 T9 评审
```

**⚠️ T9 是最后一道防线**：
如果 T9 都过了才发现问题，那就是线上事故了。所以 T9 评审必须严肃，不能走形式。发现任何 P0 问题，立即打回，绝不放水。

## 🔗 回滚路径可视化

```
T0 ──► T1 ──► T2 ──► T3 ──► T4 ──► T4.5 ──► T5 ──► T5.5 ──► T6 ──► T7 ──► T8 ──► T9 ──► 上线
│      │      │      │      │       │       │        │       │       │       │
│      │      │      │      │       │       │        │       │       │       │
◄──────┘      │      │      │       │       │        │       │       │       │
◄─────────────┘      │      │       │       │        │       │       │       │
◄────────────────────┘      │       │       │        │       │       │       │
◄───────────────────────────┘       │       │        │       │       │       │
◄───────────────────────────────────┘       │        │       │       │       │
◄───────────────────────────────────────────┘        │       │       │       │
◄────────────────────────────────────────────────────┘       │       │       │
◄─────────────────────────────────────────────────────────────┘       │       │
◄──────────────────────────────────────────────────────────────────────┘       │
◄───────────────────────────────────────────────────────────────────────────────┘
◄───────────────────────────────────────────────────────────────────────────────────────┘
```

**关键原则**：
1. **能不跨阶段回滚就不跨**（比如 T4 的问题尽量在 T4 解决，不要回 T2）
2. **越早发现问题，回滚成本越低**（T2 发现问题成本 << T6 才发现）
3. **回滚后必须从那个节点重新完整走下来**（不能跳过任何中间节点）
4. **每个节点的「完成标准」必须可量化、可验证**（不能是「感觉差不多了」）

## 📊 回滚成本统计

| 发现问题的阶段 | 平均修复成本 | 相对 T2 发现的倍数 |
|---------------|-------------|------------------|
| T2 发现 | 4 小时 | 1x |
| T3 发现 | 8 小时 | 2x |
| T4 发现 | 1 天 | 6x |
| T4.5 发现 | 2 天 | 12x |
| T5.5 发现 | 3 天 | 18x |
| T6 发现 | 5 天 | 30x |
| T9 发现 | 1 周 | 42x |
| 线上发现 | 2 周+ | 84x+ |

> 💡 **核心结论**：在 T2 多花 1 小时把设计做对，能省下 T6 的 5 天修复时间。这就是流水线的价值——把问题消灭在早期。

## 🎯 回滚机制的灵魂三问

每个节点启动前，都必须能回答这三个问题：

1. **我怎么知道上一个节点真的做完了？** → 查 DoD 完成标准
2. **如果我这里发现问题，应该打回给谁？** → 看回滚矩阵
3. **打回之后，怎么确保问题真的解决了？** → 走重入机制

如果这三个问题回答不清楚，说明这个节点的设计有缺陷，需要优化。

---

> **最终目标**：每个团队成员都清楚地知道——「我在这个节点发现问题，应该打回哪里，以及打回之后怎么再进来」。不需要每次出问题都开会讨论怎么办，流程本身就已经定义好了。
